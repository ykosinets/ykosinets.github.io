Adminka documentation

Mark any element in your HTML with data-editable and it becomes editable in place. Your client opens the page, clicks, types, and presses Save — the change is written straight back into the HTML file. No database, no JSON store, no build step: your HTML is the content.

Adminka is a single entry point (admin.php) plus a small admin-lib/ folder, dropped into the root of a static site. This page documents every feature: installation, the sign-in methods, how edit mode works, and a worked example for each editable type and content option.

Zero dependencies

Pure PHP + vanilla JS. No Composer, npm, or framework.

Files are the storage

Edits round-trip through the real HTML file, with backups.

Modern sign-in

Password, Google, Apple, and passkeys — no user database.

Requirements

  • PHP 8.1+ — 8.4+ recommended. On 8.4 Adminka uses the HTML5-accurate Dom\HTMLDocument parser; older versions fall back to legacy DOMDocument (which entity-encodes non-ASCII text — it renders fine but the source is uglier).
  • Extensions: dom (always enabled), openssl + curl for Google/Apple sign-in and passkeys, fileinfo for upload validation. All bundled by default on virtually every host.
  • HTTPS in production — required for passkeys and for the login form to be meaningful. localhost counts as secure for passkeys during development.
  • Write access for the web-server user to your HTML files and to the backups/, data/, and assets/content/ directories (created automatically).

Installation

Copy admin.php, config.php, admin-lib/, and (optionally) tools/ into the root of your static site:

/your-site
├── index.html
├── about.html
├── admin.php        ← the editor (entry point)
├── config.php       ← settings & credentials
├── admin-lib/       ← internals (PHP modules + ui.js / editor.js)
├── tools/           ← CLI helpers (assign-ids)
├── assets/content/  ← media library (image/ and video/)
├── backups/         ← created on first save
└── data/            ← passkeys + sign-in state

Make sure PHP can write to your HTML files and create those folders:

chown -R www-data:www-data /path/to/your-site
# or at minimum:
chmod 664 *.html && chmod 775 .

That is the entire install. Open https://your-site.com/admin.php and sign in.

Set your password

Never deploy with the default. The out-of-the-box login is admin / changeme — change it before going live.

Generate a hash and paste it into config.php:

php -r "echo password_hash('your-strong-password', PASSWORD_DEFAULT);"
'admin_user' => 'admin',
'admin_hash' => '$2y$10$...paste-hash-here...',

The plain password is never stored anywhere — only the bcrypt hash.

Sign-in options

Password sign-in always works and is your lockout-proof fallback. On top of it you can enable any combination of the following. Whichever method vouches for the visitor, only addresses on oauth_allowed_emails may actually sign in through OAuth.

Sign in with Google

  1. Create an OAuth client at console.cloud.google.com/apis/credentials (type “Web application”).
  2. Add the redirect URI https://your-site.com/admin.php?action=oauth_cb&provider=google
  3. Fill in config.php:
'oauth' => [
    'google' => [
        'enabled'       => true,
        'client_id'     => '1234-abc.apps.googleusercontent.com',
        'client_secret' => 'GOCSPX-...',
    ],
],
'oauth_allowed_emails' => ['you@gmail.com'],   // who may sign in

Sign in with Apple

Needs a paid Apple Developer account. Create a Services ID, enable “Sign in with Apple”, register the same redirect URI (with &provider=apple), and create a Sign in with Apple key (a .p8 file).

'apple' => [
    'enabled'   => true,
    'client_id' => 'com.your-site.adminka',   // the Services ID
    'team_id'   => 'ABCDE12345',
    'key_id'    => 'XYZ9876543',
    'key_file'  => __DIR__ . '/AuthKey.p8',   // keep out of web root if possible
],

Identity tokens are validated for issuer, audience, expiry, and a per-attempt nonce before the email is checked against the allowlist.

Passkeys no config

Touch ID / Face ID / Windows Hello / security keys — nothing to configure. Sign in once (any method), then click “Add a passkey for this device” on the page list. From then on the login screen offers “Sign in with a passkey.”

  • Requires HTTPS (or localhost).
  • Credentials live in data/passkeys.json; use the Remove button (or delete a line) to revoke one.
  • ES256 and RS256 authenticators are supported — Apple, Google, Windows Hello, YubiKeys.

Core concepts

Two attributes drive everything:

  • data-editable — marks an element as editable. Its value is a per-page-unique id (auto-generated if you leave it blank; see Auto-assigned IDs).
  • data-editable-type — chooses the editing behavior (text, html, image, video, link, list, form, attrs). Omit it for plain text.

The type is read from the file on the server, never trusted from the browser — so a tampered client cannot, say, turn a text field into raw HTML. When your client presses Save, the changed values are applied to the parsed document and written back atomically (temp file + rename), after a timestamped backup.

Edit mode is a canvas

When a page is opened through admin.php?page=…, it renders exactly like the live site with editable regions outlined — but the page’s own JavaScript and navigation are frozen. Links don’t navigate, anchors don’t scroll, forms don’t submit, and the site’s menus/sliders/carousels don’t react. This keeps the editor stable no matter what scripts the page ships.

Because page scripts are frozen, widgets that rely on JS to reveal content (sliders, popups, accordions) need a marker so their content stays reachable while editing — see Sliders and Popups & accordions.

Auto-assigned IDs

You never have to invent ids. Write just the marker:

<h1 data-editable>We build websites</h1>
<img src="team.jpg" data-editable-type="image">

The first time a page is opened in the admin (or saved through it), every editable without an id gets a readable one — slugged from the element’s text (we-build-websites), transliterated from non-Latin scripts, filename-based for images/videos, class-based for lists and forms, de-duplicated with numeric suffixes. The file is normalized once, with a backup, and stays stable after. Duplicate ids are healed the same way; hand-written ids are never touched.

To normalize files before deploying (avoiding a first-open write in production), run the CLI tool.

text

data-editable-type="text"default

Plain-text, inline editing on any element. HTML the client types is escaped, so it can never inject markup. Source indentation is shown collapsed in the editor and text is trimmed on save, so IDE whitespace never leaks into the content.

The panels below are live and split in two: the Admin view is what your client edits, the Production view is what visitors see. Edit on the left, press Save changes, and the right side updates — exactly the adminka round-trip. Reset restores a demo.

  We build websites that just work
</h1>

html

data-editable-type="html"

Rich, inline editing for a block of formatted content. On save the markup is sanitized against a tag/attribute whitelist (configured in config.php): script, style, iframe, object, and form are removed with their content, event handlers are stripped, and disallowed tags are unwrapped.

Markup 
<div data-editable="about-body" data-editable-type="html">
  <p>We are a <strong>small studio</strong> focused on fast sites.</p>
  <p>No heavy CMS — just clean HTML.</p>
</div>

Default allowed tags: p br b strong i em u s a ul ol li h1–h6 blockquote span. Default allowed attributes: a[href,title,target], span[class].

image

data-editable-type="image"

Clicking an <img> opens the media picker: choose an existing file, upload a new one (with an optional crop step for JPEG/PNG/WebP), or paste a URL. An Alt text… button edits the alt attribute. Images buried under overlay text also get a 🖼 corner badge so they stay clickable.

Markup 
<img src="assets/content/image/hero.jpg" alt="Storefront"
     data-editable="hero-image" data-editable-type="image">

javascript: and data: URLs are rejected. Uploads are validated by extension, sniffed MIME type, and size cap.

video
data-editable-type="video"

      
Clicking a <video> opens a dialog rather than the bare picker. You can set the video file and a poster image (each via Browse… into the library), toggle Show player controls, and toggle Autoplay silently in the background — which applies the reliable background-video recipe in one click.

      Markup
      
<video src="assets/content/video/reel.mp4" controls
       data-editable="showreel" data-editable-type="video"></video>

      What “background autoplay” writes
      
<video src="…" muted autoplay loop playsinline webkit-playsinline
       disablepictureinpicture disableremoteplayback preload="metadata"></video>

      
A 🎬 corner badge is added for background videos that content sits on top of.

list
data-editable-type="list"

      
Turns a container into a repeatable list. In edit mode a “+ Add item” tile appears; each item gets a “×” remove badge (hidden when only one item remains, since new items are copied from the last one). Add and remove save immediately, with a backup, and the server assigns fresh unique ids to the new item’s nested editables automatically.

      Markup
      
<div class="cards" data-editable="services" data-editable-type="list">
  <div class="card">
    <img src="a.jpg" data-editable-type="image">
    <h3 data-editable-type="text">Landing pages</h3>
  </div>
  <!-- more .card items… -->
</div>

      
Nested editables don’t need ids — they’re generated on first open. For a horizontally-scrolling gallery, combine with data-editable-scroll.

form
data-editable-type="form"

      
Mark a <form> (or any wrapper around controls). In edit mode every input, textarea, select, and button becomes click-to-edit. The editor stays deliberately content-focused: label, placeholder (only for input types that support one), and required; buttons get their text. Structural details — field name, input type, default value — stay in the markup where the developer put them.

      Markup
      
<form data-editable="contact-form" data-editable-type="form">
  <label for="cf-name">Your name</label>
  <input id="cf-name" name="name" type="text" placeholder="Jane Doe" required>
  <select name="topic">
    <option>New website</option>
    <option value="other">Something else</option>
  </select>
</form>

      
Select options are a simple label list: rename inline, drag ☰ to reorder, × to remove, + to add. Each option’s value/selected/disabled travel with it untouched — the editor changes only the visible words. See also the select placeholder.

attrs
data-editable-attrs="…"

      
Expose specific attributes for editing. Hovering the element shows a ⚙ button that opens an attribute dialog — untick to remove an attribute, tick with an empty value for boolean attributes like download or controls.

      Markup
      
<a href="/kit.zip" download
   data-editable="dl" data-editable-type="attrs"
   data-editable-attrs="href download title">Download the kit</a>

      
    
        
  • Only attributes named in data-editable-attrs can be changed (the server re-reads the list from the file).
    • 
        
  • Event handlers, style, and data-editable* are always refused; URL-carrying attributes (href, src, poster, …) are scheme-validated.
    • 
        
  • Use type attrs when attributes are the only thing to edit; or combine data-editable-attrs with any other type to get both.
    • 
      

      
If you list src on an element the media picker also manages, the picker’s value wins on save — simplest not to double-manage the same attribute.

      

        
Live demo — hover the button, click the ⚙ that appears

Sliders
data-editable-scroll

      
Since page JS is frozen in edit mode, a carousel would be stuck on its first slide. Add data-editable-scroll to the slider’s track and, in edit mode only, it becomes a horizontal scroll-snap strip — every slide reachable by scrolling, no arrows needed. The live page is untouched.

      Markup
      
<div class="slider" data-editable="gallery"
     data-editable-type="list" data-editable-scroll>
  <figure> … </figure>
  <figure> … </figure>
</div>

      
Combine with data-editable-type="list" (as above) so the client can add and remove slides too.

Popups & accordions
data-editable-reveal

      
Content the live site keeps hidden — a modal, a collapsed accordion panel, an off-screen drawer — would be invisible (and uneditable) with page JS frozen. Mark it data-editable-reveal and it’s shown inline while editing, flagged “Hidden on the live site,” with everything inside editable as normal. The trigger button/link is just a regular text or link editable.

      Markup
      
<div class="faq-item">
  <button class="faq-q" data-editable="q1" data-editable-type="text">
    How long does a site take?
  </button>
  <div class="faq-a" style="display:none" data-editable-reveal>
    <p data-editable="a1" data-editable-type="text">About a week.</p>
  </div>
</div>

      
Adminka can’t guess which elements are popups — it reveals only what you mark. Bake the marker into your reusable popup/accordion partials once and it’s free on every page.

      

        
Live demo — the panel is display:none live, revealed for editing

Media library

      
Images live in assets/content/image, videos in assets/content/video (configurable). The picker shows everything in the matching folder, newest first, plus:

      
    
        
  • Upload… — the file lands in the folder and is selected immediately. JPEG/PNG/WebP get a crop dialog first (drag the corners, then Crop & upload — re-encoded at full resolution in the browser — or Upload original). GIF and AVIF skip cropping so animation/transparency survive.
    • 
        
  • Use URL… — reference an external URL instead.
    • 
        
  • Alt text… — edit an image’s alt.
    • 
      

      
Every upload is checked three ways: extension whitelist, sniffed MIME type must match the folder kind, and a size cap (10 MB images / 200 MB videos by default). SVG uploads are refused (SVG can carry scripts). Filenames are slugified; collisions get -2, -3 suffixes.

      

        
KindFolderExtensionsMax size

        
imageassets/content/imagejpg, jpeg, png, gif, webp, avif10 MB

        
videoassets/content/videomp4, webm, mov200 MB

      

    


    

      
Select placeholder

      
A <select>’s control editor has a Placeholder field. Typing one creates (or updates) a hidden, disabled first option — the standard “please choose” pattern — which is excluded from the draggable option rows so it never clutters the list. Clearing the field removes it.

      What the placeholder field writes
      
<option value="" disabled selected style="display:none;">
  Pick a topic…
</option>

Backups & recovery

      
Before every save the current file is copied into backups/ with a timestamp. Adminka keeps the last 10 per file (set by backup_keep) — the oldest is pruned before each new one is written, so the count never grows past the cap.

      
backups/index.html.2026-07-12_142530

      
To restore, copy a backup over the live file:

      
cp backups/index.html.2026-07-12_142530 index.html

      
Writes are atomic (temp file + rename), so a failed save can never leave a half-written page.

    


    

      
Security notes

      
Built in: bcrypt password + PHP sessions, a brute-force delay, a CSRF token on every state-changing request (saves, uploads, passkey management), path confinement to site_root (no ../ traversal), server-side type enforcement, an HTML whitelist sanitizer, URL-scheme validation for images/videos/links, upload MIME sniffing, OAuth state/nonce validation with an email allowlist, and WebAuthn signature + counter verification (cloned-authenticator detection).

      
Your responsibilities:

      
    
        
  • Serve the site over HTTPS.
    • 
        
  • Change the default password before going live.
    • 
        
  • Optionally add a second layer in front of admin.php (HTTP Basic Auth or an IP allowlist).
    • 
        
  • Deny web access to backups/, data/, config.php, and your Apple key:
    • 
      

      
# nginx
location ~ ^/(backups/|data/|config\.php|AuthKey) { deny all; }

      
On Apache, data/ gets a deny-all .htaccess automatically; add the same file to backups/.

    


    

      
CLI: assign-ids

      
ID generation also runs automatically on first open, but you can normalize files ahead of a deploy (to avoid a first-open write in production):

      
php tools/assign-ids.php index.html            # fills ids, keeps index.html.bak
php tools/assign-ids.php index.html --dry-run  # preview only

      
It generates readable slugs, transliterates non-Latin text, de-duplicates, never touches existing ids, and reports any duplicates it heals.

    


    

      
Troubleshooting

      

        
SymptomFix

        
“Could not write file. Check permissions.”Give the web-server user write access to the HTML files and site directory.

        
“Invalid CSRF token — reload the page.”Session expired; reload the edit page and save again.

        
Page list is emptyCheck site_root and extensions in config.php.

        
Element not editableVerify the data-editable id is present and unique on the page.

        
“Sign-in session expired” during OAuthThe state file expired (10 min) or was reused — start sign-in again.

        
Passkey button missing on loginNo passkeys registered yet, or data/passkeys.json unreadable.

        
Passkey registration fails instantlyPage must be served over HTTPS (or localhost); check the browser console.

        
Upload rejected as wrong typeContent is sniffed — renaming .mov→.mp4 is fine, but a PHP script named .png is not.

        
Slider/menu/popup doesn’t react in edit modeBy design — page JS is frozen. Mark sliders with data-editable-scroll, hidden panels with data-editable-reveal.

        
Styles broken in edit mode for subfolder pagesUse root-relative asset paths (/css/style.css) instead of relative ones.
